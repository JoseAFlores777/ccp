ccp handoff — continuar una sesión en otro perfil sin perder contexto

Te quedas sin tokens a media conversación. ccp handoff copia el contexto al cc-home de otro perfil, reanuda ahí con tokens frescos, y al volver trae el trabajo de regreso como una sesión nueva — sin destruir nada.

§ 01Overview

La restricción dura primero, luego el camino real.

Cuando trabajas en una sesión de Claude Code con un perfil ccp y ese perfil se queda sin tokens, hoy no hay forma limpia de seguir con otro perfil sin perder el contexto. La restricción dura: un proceso claude vivo congeló sus credenciales al arrancar; ccp use solo reescribe el env del shell y un proceso vivo nunca lo re-lee. No existe hot-swap de credenciales.

El camino real: persistir el contexto → cambiar de perfil → reanudar la misma conversación en un proceso nuevo, con tokens frescos del destino. El contexto vive en disco como transcript JSONL. ccp handoff lo automatiza y deja un marcador para el viaje de vuelta.

El modelo mental: pides prestados los tokens de otro perfil para una sesión, y devuelves el trabajo al volver. ccp handoff end trae el contexto actualizado de regreso al perfil origen como una sesión nueva (no destructiva) y reanuda ahí.

§ 02Goals / Non-goals

Goals

  • Continuar una conversación bajo otro perfil sin perder contexto, con tokens del destino.
  • Flujo TUI: elegir perfil destino (excluyendo el activo) → elegir sesión (nuevo→viejo, con fecha/título/uuid) → confirmar marcador.
  • Equivalente CLI por flags (ccp handoff <to> --session <uuid>) para no-TTY/scripting.
  • Marcador de referencia en el perfil origen para recordar el handoff en vuelo.
  • ccp handoff end: back-sync destino→origen como sesión nueva (uuid nuevo), no destructivo, y relanzar en origen.
  • Origen visible: la sesión de vuelta muestra [de <perfil>] en su título, en la TUI de ccp y en el menú nativo claude --resume.
  • No confundir a Claude Code: la sesión reescrita reanuda limpia (validado en disco, §08).

Non-goals

  • No hot-swap de credenciales en un proceso vivo (imposible).
  • No handoff encadenado multi-nivel: v1 permite 1 nivel y bloquea un segundo handoff hasta hacer end. Cadena real = v2.
  • No copiar blobs de file-history/ (rewind de archivos): la conversación reanuda perfecta, solo ese feature degrada. v2.
  • No handoff entre máquinas o cwd distintos: asume misma máquina, mismo repo.
  • No auto-detección de "volvieron los tokens": end es manual.
  • No funcionar en cloud/headless sin ccp: es para máquina local con ccp, shell function y TTY.

§ 03Arquitectura y superficie de comandos

Encaja en el split central de ccp sin cambiarlo: lógica en el binario, mutación de env/launch en la shell function.

Comandos de cara al usuario

# interactivo y por flags
ccp handoff                          # TUI: pick perfil → pick sesión → confirma
ccp handoff <to>                     # salta picker de perfil
ccp handoff <to> --session <uuid>    # salta ambos pickers (scriptable)
ccp handoff <to> --no-marker         # forward sin marcador (warning)
ccp handoff <to> --force             # fuerza sobre colisión de uuid
ccp handoff end                      # back-sync destino→origen, relanza en origen
ccp handoff status                   # handoff activo (o "sin handoff activo")
ccp handoff list                     # historial + activo

Comandos internos (emiten env, estilo _env/_hook)

command ccp _handoff <to> <pwd>      # forward: copia + marcador + emite env(destino) + CCP_RESUME_ID
command ccp _handoff-end <pwd>       # return: back-sync + archiva + emite env(origen) + CCP_RESUME_ID

El split en la shell function

handoff)
  case "$2" in
    end)          out=$(command ccp _handoff-end "$PWD") || return ;;
    status|list)  command ccp handoff "$2"; return ;;   # solo lectura
    *)            out=$(command ccp _handoff "${2:-}" "$PWD") || return ;;
  esac
  ( eval "$out"; claude --resume "$CCP_RESUME_ID" )   # subshell → env efímero, como `run`
  ;;

El eval corre en subshell para que el env del destino no persista tras salir de claude (decisión: efímero, como run). El binario hace toda la lógica; solo emite el env delta + CCP_RESUME_ID.

Componentes nuevos

internal/core/handoff.go internal/tui (2 pickers) internal/cli (dispatch) core/shellinit.go (case handoff)

§ 04Modelo de datos — el marcador

Estado de runtime, no config. Archivo nuevo ~/.config/ccp/handoffs.yaml, escritura atómica bajo flock (patrón de store.go).

No se mezcla con ccp.yaml (config declarativa versionada): mezclarlos ensuciaría el diff de config y arriesgaría el version: 2 del store. Si handoffs.yaml se corrompe, no rompe perfiles ni reglas — degradación suave; los jsonl siguen en disco.
version: 1
active:                              # 0 o 1 entradas (regla "1 nivel v1")
  session: bbc1ed61-ada1-408f-...    # uuid copiado (mismo en origen y destino)
  slug: -Volumes-...-dsctl-v2        # proyecto (cwd → guiones)
  cwd: /Volumes/.../dsctl-v2         # ruta absoluta, para validar
  from: personal-cc                  # perfil padre
  to: emco-cc                        # perfil destino (tokens prestados)
  title: "Refactor handoff en ccp"   # aiTitle al forward (display)
  since: 2026-06-19T14:30:00Z
archived:                            # historial (para `handoff list`)
  - session: 9c2e0d4f-...
    from: personal-cc
    to: emco-cc
    returned_as: a1b2f0d3-...        # uuid nuevo del back-sync
    since: 2026-06-18T10:00:00Z
    ended: 2026-06-18T15:20:00Z
  • active presente → hay handoff en vuelo. Un segundo handoff → error 1-nivel.
  • end → toma active, back-sync, mueve a archived con returned_as+ended.
  • Migración: ninguna. Archivo nuevo, nace en version: 1.

§ 05Flujo forward (ccp handoff)

  1. Pre-chequeos: ¿hay active? → error 1-nivel. Origen = perfil activo (CLAUDE_CONFIG_DIR o Resolve(cwd)); default permitido.
  2. Picker de perfil destino (TUI) o <to>: lista perfiles menos el activo. destino==origen o inexistente → error.
  3. Picker de sesión o --session: escanea origen/projects/<slug>/*.jsonl; lee mtime, aiTitle, uuid; ordena nuevo→viejo. Carpeta vacía → error.
  4. Confirmar marcador ([Y/n]; --no-marker salta): default sí.
  5. Copia forward (mismo uuid, copia cruda): sessionId ya == uuid, cwd mismo repo → sin reescritura. Colisión mismo-contenido → overwrite+warn; distinto → error sin --force.
  6. Warning cross-provider si origen/destino difieren en tipo o modelo.
  7. Escribe marcador + emite env: EnvDelta(destino) + CCP_RESUME_ID=<uuid>.
  8. Shell: eval (subshell) + claude --resume.
Orden transaccional: copia → marcador → emit env. Copia falla → sin marcador huérfano. Marcador falla → no se emite env → no se lanza claude → estado consistente (jsonl copiado pero no marcado; recuperable).

§ 06Flujo return (ccp handoff end) y reescritura

Lo más delicado: el back-sync no es cp — procesa el jsonl línea por línea.

  1. Pre-chequeos: ¿hay active? No → error. cwd ≠ active.cwd → warning. ¿Existe el jsonl en destino? No → error (marcador queda activo).
  2. Genera uuid nuevo N con crypto/rand (UUID v4).
  3. Reescritura de destino/.../<uuid_viejo>.jsonl:
    • sessionId → N (todas las líneas).
    • type:"ai-title" → prepende [de <to>] (idempotente).
    • NO toca: cwd, árbol uuid/parentUuid/leafUuid, messageId, timestamps, contenido.
    • escribe en origen/.../N.jsonl.
  4. No-destructivo: original en origen intacto; el de destino tampoco se borra (solo se lee). Origen termina con dos sesiones: la vieja + N completa.
  5. Validación: re-parsea N.jsonl; assert cero sessionId viejo y JSONL válido. Falla → borra N.jsonl, no archiva, error.
  6. Archiva marcador + emite env del origen + CCP_RESUME_ID=N.
  7. Shell: eval (origen) + claude --resume N.

§ 07Manejo de errores y casos borde

Principio: toda operación es transaccional o no pasa nada.

CasoComportamiento
Handoff activo, intentas otroError 1-nivel: termina el activo con ccp handoff end.
end sin handoff activoError: sin handoff activo.
destino == origenError no-op.
destino no existe como perfilError + lista perfiles válidos.
--session <uuid> inexistenteError + lista sesiones del cwd.
Carpeta projects/ vacíaError: sin sesiones para este proyecto.
Colisión uuid, mismo contenidoOverwrite + warning.
Colisión uuid, contenido distintoError, pide --force.
Cross-provider (Anthropic→DeepSeek)Warning no-bloqueante.
jsonl de destino borrado antes de endError; marcador queda activo.
Reescritura produce JSONL inválidoBorra N.jsonl, no archiva, error (estado pre-end).
cwd en end ≠ marcadorWarning, permite.
handoffs.yaml corrupto/ausenteDegradación suave; forward arranca limpio.
Usuario nunca corre endMarcador activo; status lo recuerda.
default como origen o destinoPermitido (cc-home = ~/.claude).
Concurrencia (dos handoff)flock serializa; el segundo → error 1-nivel.

Exit codes (estable): forward/end OK → 0; pre-chequeo → 1; I/O → 2; handoff status → 0 si activo, 1 si no (paridad con resolve).

§ 08Formato en disco de Claude Code (validado)

Investigado en disco (CC v2.1.183). Sesión en <CLAUDE_CONFIG_DIR>/projects/<slug>/<uuid>.jsonl, una línea JSON por evento.

Identidad de sesión (mantener consistente al copiar)

  • El uuid del nombre de archivo ES el id para claude --resume <uuid>.
  • El campo sessionId (~90% de líneas) debe igualar el uuid del archivo. Copiar como sesión nueva exige reescribir cada sessionId (no es cp crudo).
  • cwd (ruta absoluta) → misma máquina/repo, sin cambios.
  • Título = aiTitle en líneas type:"ai-title" (esta versión no usa type:"summary").
  • Árbol uuid/parentUuid/leafUuid (last-prompt guarda el leaf) — interno, intacto.

NO es identidad — no tocar, sin riesgo

  • .claude.json — llaveado por cwd, solo guarda lastSessionId+métricas, no enumera sesiones; resume-by-uuid escanea la carpeta directo.
  • session-env/<sid>/ — dir scratch vacío, sin credenciales (cero leak).
  • sessions/<pid>.json — registro de proceso vivo por PID, efímero.
Degradación residual: file-history/ (rewind/restore) llaveado aparte y no se copia → la conversación reanuda perfecta, solo el rewind de archivos puede no hallar blobs. Documentado; copia opcional en v2.

§ 09Estrategia de pruebas

Feature nuevo (fuera del contrato bash congelado), pero la shell function gana el case handoff) — texto byte-idéntico en shellinit.go ⟷ oráculo bash.

Gates existentes (obligatorio)

  • Añadir handoff a la shell function → actualizar oráculo bash en legacy/ + regenerar golden con capture.sh + parity_test.go verde.
  • _handoff/_handoff-end emiten env → al fixture golden como comandos de scripting.
  • gofmt/vet/golangci-lint limpios.

Unitarios (internal/core/handoff_test.go)

  • Reescritura jsonl (corazón): sobre fixture real, assert sessionId viejo→N en todas las líneas (cero residuos); aiTitle prepende e idempotente; cwd/árbol/messageId/timestamps/contenido intactos; salida JSONL válida.
  • Marcador CRUD: write→read→archive; active único; atómico bajo flock.
  • Copia forward: mismo uuid; colisión mismo-contenido vs distinto.
  • Listado: orden nuevo→viejo, extracción de aiTitle, carpeta vacía → error.
  • Resolución de origen: desde CLAUDE_CONFIG_DIR y Resolve(cwd).

Eval-effect (zsh + bash) y E2E

eval "$(ccp _handoff ...)" exporta CLAUDE_CONFIG_DIR(destino)+CCP_RESUME_ID; _handoff-end el del origen + N. E2E con temp CCP_HOME (dir existente, no dispara auto-migración): forward → assert copiado + marcador + env; end → assert N.jsonl reescrito, original intacto, marcador archivado. Nunca lanza claude (el binario solo emite env).
