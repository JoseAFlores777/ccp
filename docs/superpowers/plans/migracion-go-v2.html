§ 00Objetivo e invariantes

Migrar ccp de Bash a Go sin que el usuario pierda data ni configuración al actualizar. Un ccp upgrade (todavía bash en ese momento) deja instalado el binario Go; el siguiente command ccp ya es Go y migra los datos a YAML de forma respaldada. El bloque rc no se toca.

Invariantes que NO pueden romperse (contrato congelado)

  1. ccp _env <perfil> y ccp _hook <path> se hacen eval en el shell → salida equivalente byte-a-byte al bash (mismo unset/export, mismo quoting, mismo orden).
  2. ccp completion-shellinit produce el bloque rc entre # >>> ccp shell init >>> / # <<<.
  3. ccp completion zsh|bash produce completions válidos (el rc los eval al arrancar).
  4. Superficie scriptable estable: resolve, path test (exit 0/1), status --json.
  5. Frontera binario↔shell: el binario EMITE entorno, el shell lo EVALúa. Nunca el binario muta el entorno.

§ 02Arquitectura objetivo

Core en un paquete Go; dos front-ends (CLI y TUI) sobre el mismo core. La TUI nunca duplica lógica.

Module: github.com/JoseAFlores777/ccp · binario ccp · Go mínimo 1.23. El repo se renombra dsctl → ccp (alinea repo=binario=module; coste: re-apuntar install-source, redirect de GitHub).

ccp/
├── cmd/ccp/main.go              # entrypoint: parse args -> dispatch (a mano, sin cobra)
├── internal/
│   ├── core/                    # motor (sin I/O de presentación)
│   │   ├── profile.go           # CRUD perfiles (official/deepseek/default)
│   │   ├── rules.go             # path rules, resolve (deepest wins)
│   │   ├── env.go               # EnvDelta: unset/export %q-equivalente
│   │   ├── cfg.go               # overlay engine (@import + jq-like merge)
│   │   ├── store.go             # lee/escribe ccp.yaml (atómico)
│   │   ├── secrets.go           # api_key 600 + credenciales cc-home
│   │   ├── backup.go            # export/restore tar.gz + manifest
│   │   ├── migrate.go           # migrador universal dsctl->ccp->YAML
│   │   └── shellinit.go         # cola shell bash como string
│   ├── cli/                     # dispatch subcomandos, salida texto/JSON
│   └── tui/                     # bubbletea app + formularios huh
├── testdata/golden/             # fixtures CCP_HOME + salidas esperadas
├── install.sh                   # detecta OS/arch: release o go build
├── legacy/                      # bash archivado tras paridad
└── .github/workflows/           # build matrix + release + golden gate
Regla dura. internal/core no imprime presentación; devuelve datos/strings y los front-ends formatean. Solo env.go y shellinit.go producen strings exactos — son el contrato.

§ 03Contrato congelado — equivalencia

El punto fino es el quoting. Bash printf '%q' quotea de forma específica (p.ej. $'...' para caracteres de control). Go debe producir algo que zsh y bash evalúen a la misma cadena.

  • Implementar core.shellQuote(s string) string que replique %q de bash para valores reales (paths con espacios, base_url, tokens, modelos). No usar strconv.Quote (semántica Go ≠ shell).
  • Verificación por efecto: eval la salida de Go en un zsh y un bash reales, y comparar las variables resultantes contra el bash oráculo — no solo el texto.
  • Orden idéntico: primero unset de todas las managed vars, luego los export del perfil objetivo.

§ 04Modelo de datos — ccp.yaml

Fuente de verdad única. Reemplaza profiles.tsv + rules.tsv + config + los meta + el authored global/profile. Secretos fuera.

version: 2                      # si el binario conoce < version -> ABORTA

defaults:                       # SOLO siembra perfiles nuevos; editar no muta existentes
  base_url: https://api.deepseek.com/anthropic
  model_pro: deepseek-chat
  model_flash: deepseek-chat
  effort: high
  editor: nano

profiles:                       # 'default' es IMPLÍCITO (nunca aquí)
  work:
    type: official             # login vive en cc-home/.claude.json
  deepseek:                     # literal load-bearing (env.go switchea sobre él)
    type: deepseek
    base_url: https://api.deepseek.com/anthropic  # 4 campos EXPLÍCITOS
    model_pro: deepseek-chat
    model_flash: deepseek-chat
    effort: high
    # api_key NO va aquí -> profiles/deepseek/api_key (chmod 600)

rules:                          # lista {path, profile}. path abs único. deepest-wins runtime
  - path: /Users/me/work
    profile: work

authored:                       # SOLO global+profile, machine-local
  - scope: global
    type: rule
    ref: "..."
    desc: "..."

Reglas del schema

  • Versión: entero; binario con versión conocida menor a la del archivo → aborta, no escribe.
  • Claves desconocidas: preservadas en round-trip (catch-all inline) — un ccp viejo no borra campos de uno nuevo.
  • Comentarios: preservados vía CommentMap de goccy (decode por-path → re-adjunta en encode).
  • Sin herencia: deepseek guarda sus 4 campos completos; defaults solo siembra al crear.
  • Escritura atómica: tmp + rename. default nunca se serializa.
Fuera del YAML (igual en disco): cc-home/, overlay/, api_key (600), credenciales OAuth, install-source, y el authored project .claude/ccp-authored.tsv — ccp no reescribe archivos del repo del usuario (cero churn de git). Asimetría a propósito.

§ 05Migración de datos (migrador universal)

Disparado en el primer arranque Go cuando detecta estado legacy. Atómico e idempotente.

detectar estado:
  ~/.config/dsctl && !~/.config/ccp  -> migrar dsctl -> ccp (TSV) -> seguir
  ~/.config/ccp con *.tsv/meta sin ccp.yaml -> migrar TSV/meta -> ccp.yaml
  ccp.yaml existe -> no-op
antes de tocar nada:
  copiar estado legacy -> ~/.config/ccp/.backup-pre-go-<fecha>/
migrar:
  profiles.tsv + meta -> profiles ; rules.tsv -> rules
  config -> defaults ; authored.tsv -> authored
  escribir ccp.yaml (atómico)
  NO borrar api_key ni cc-home
Puerta de una vía. Tras migrar, el bash viejo no lee YAML. Es recuperable: el .backup-pre-go-* conserva el TSV/meta; rollback = reinstalar bash + restaurar ese backup. Ver ADR 0007.

§ 06Backup / restore

Export — ccp backup export [archivo.tar.gz] [--with-secrets]

  • .tar.gz (preserva permisos 600) con manifest.yaml (versión, fecha, perfiles, flag with_secrets, checksum) + ccp.yaml + overlay/ de cada perfil.
  • Con --with-secrets: añade api_key + credenciales del cc-home; archivo chmod 600 + warning.
  • Excluye symlinks re-seedables (plugins/commands/agents/skills) — se reconstruyen al restaurar.

Restore — ccp backup restore archivo.tar.gz [--overwrite | --force]

verificar checksum del manifest
auto-snapshot estado actual -> .backup-pre-restore-<fecha>/   # reversible
por cada perfil del backup:
  no existe        -> crear (re-seed symlinks cc-home)
  existe (default) -> skip + reportar
  --overwrite      -> reemplazar ese perfil
  --force          -> wipe + restore total
reglas de path: merge (las que no chocan se añaden)
secretos (si full) -> escribir chmod 600

§ 07TUI — bubbletea + huh

ccp sin args y con TTY lanza la TUI. Sin TTY (pipe/script) imprime ayuda/estado.

Perfiles

Lista navegable; enter → detalle. Acciones vía huh: añadir (official/deepseek), editar config ($EDITOR), login, borrar (confirm), set key.

Reglas

Lista de path rules; añadir (huh: path + select de perfil), borrar (confirm).

Estado

Perfil del cwd resuelto en vivo, perfil activo de la terminal, repo.

Barra de comandos

backup export/restore (wizard huh), doctor, sync, install. Navegación: tab entre paneles, j/k dentro.

Sin duplicar lógica. La TUI solo llama internal/core; cada acción tiene su subcomando CLI equivalente. Degrada con isatty/NO_COLOR.

§ 08Distribución, install, upgrade

CI: matriz darwin/arm64·amd64, linux/amd64·arm64 → binarios + checksums.txt en GitHub Releases.

# install.sh (Go-aware)
detectar OS/arch
release disponible -> descargar binario + verificar checksum -> ~/.local/bin/ccp
si no, hay toolchain Go -> go build ./cmd/ccp
rm -rf ~/.local/lib/ccp        # ya no hay libs bash
registrar install-source ; copiar commands/ccp/*.md
El corte bash→Go. Usuario en bash corre ccp upgrade → re-ejecuta el install.sh nuevo → instala binario Go y borra libs bash. El bloque rc no cambia (solo llama command ccp), así el siguiente command ccp ya es Go, y la migración-a-YAML dispara en ese primer arranque.
Nota operativa. En la máquina actual Go no está instalado → el fallback go build no aplica; el camino prebuilt-release es el efectivo. El binario debe instalarse sin toolchain Go.

§ 09Estrategia de tests

  • go test (core): tablas para profile, rules (deepest-wins, norm-path), env (EnvDelta + quoting), cfg (overlay merge), store (round-trip YAML), backup (export/restore + colisiones), migrate (cadena + idempotencia).
  • Golden-diff (gate CI): fixtures testdata/golden/<caso>/CCP_HOME. Correr binario bash (oráculo) y Go; exigir salida idéntica en _env, _hook, completion*, status --json, resolve, path test — más el efecto del eval en zsh y bash reales.
  • tests/run.sh se mantiene mientras el bash es oráculo; se retira al lograr paridad.

§ 12Criterios de aceptación (v2.0)

  1. Golden-diff verde: salida y efecto del eval de toda la superficie idénticos al bash, en zsh y bash.
  2. ccp upgrade desde una instalación bash real deja el binario Go, migra a ccp.yaml con backup, y cd/claude siguen funcionando sin reinstalar el rc.
  3. Migración idempotente y reversible (el .backup-pre-go-* permite volver a bash).
  4. backup export/restore round-trip completo (config y full), restore no-destructivo con snapshot previo.
  5. TUI lanza con ccp (TTY), cubre CRUD perfiles/reglas + backup/doctor, no rompe el modo no-interactivo.
  6. install.sh instala por release (fallback go build) en darwin/linux × arm64/amd64.
  7. Secretos nunca en ccp.yaml; backups sin --with-secrets libres de secretos.
