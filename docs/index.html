dsctl

Alterna Claude Code entre tu login oficial de Anthropic y los modelos de DeepSeek — controlado por terminal y por carpeta con reglas include / exclude.

Las variables ANTHROPIC_BASE_URL / ANTHROPIC_AUTH_TOKEN que usa DeepSeek anulan tu login. Si las pones globales, contaminan todas tus sesiones — incluidas las de empresa. dsctl las activa solo donde tú definas, nunca de forma global, así tu login oficial queda intacto.

§ 01

Instalación

Requisitos: Node.js 18+ y Claude Code. Compatible con bash y zsh.

$ git clone https://github.com/<tu-usuario>/dsctl.git
$ cd dsctl
$ bash install.sh          # copia binario y librería a ~/.local
$ dsctl install            # añade la función 'ds' + hook + completion a tu shell
$ source ~/.zshrc          # o ~/.bashrc
$ dsctl key                # guarda tu API key DeepSeek (entrada oculta)

Instala Claude Code con npm install -g @anthropic-ai/claude-code. Verifica todo con:

$ dsctl doctor
¿Por qué dos pasos? install.sh copia el binario; dsctl install escribe la función de shell en tu rc. La función es la única que puede modificar el entorno de tu terminal — el binario corre en un proceso hijo y no puede. Ver Cómo funciona.
§ 02

Inicio rápido

El flujo más común: marca una carpeta como territorio DeepSeek y olvídate. Al entrar, Claude Code usa DeepSeek; al salir, vuelve a tu login oficial — automático.

$ dsctl path include ~/work     # bajo ~/work → DeepSeek
✓ INCLUDE: DeepSeek activo en  /Users/tu/work  (y subcarpetas)

$ cd ~/work/proyecto
🟢 DeepSeek ACTIVADO en esta terminal (deepseek-v4-pro[1m])

$ claude                         # corre con DeepSeek

$ cd ~                           # fuera de la regla
⚪ DeepSeek DESACTIVADO. Claude Code usará tu login oficial.

¿Prefieres control manual sin reglas? Usa ds on / off / run en la terminal actual.

§ 03

Reglas de PATH

El núcleo de dsctl. Defines dónde corre DeepSeek con dos tipos de regla:

ReglaEfecto
includeBajo este path (y subcarpetas) DeepSeek se enciende.
excludeBajo este path nunca, aunque herede un include.

Precedencia

Gana la regla más específica (la ruta más profunda). En empate exacto, exclude gana. Sin reglas que apliquen → Claude Code oficial.

$ dsctl path include ~/work              # todo ~/work usa DeepSeek
$ dsctl path exclude ~/work/cliente      # ...menos este cliente
$ dsctl path include ~/work/cliente/lab  # ...pero su carpeta 'lab' sí

Resolución resultante:

~/work 🟢 DeepSeek include
└ ~/work/cliente ⚪ oficial exclude — gana sobre el include padre
└ ~/work/cliente/lab 🟢 DeepSeek include más profundo — re-activa
└ ~/otra-cosa ⚪ oficial ninguna regla aplica

Administrar reglas

$ dsctl path list             # árbol de includes/excludes + regla del cwd
$ dsctl path test ~/work/x    # ¿esta ruta resolvería a DeepSeek u oficial?
$ dsctl path rm ~/work/cliente # quita la regla de esa ruta exacta
$ dsctl path clear            # borra TODAS las reglas
$ dsctl path edit             # abre rules.tsv en $EDITOR
Omite la ruta y dsctl usa la carpeta actual: dsctl path include incluye el cwd.
§ 04

Uso por terminal

Control manual, sin tocar reglas. Afecta solo la terminal actual.

$ ds on            # 🟢 DeepSeek SOLO en esta terminal
$ claude
$ ds off           # ⚪ vuelve al login oficial
$ ds run           # enciende, corre claude, restaura al salir
$ ds run npm test  # enciende, corre cualquier comando, restaura
ComandoQué hace
ds onExporta las variables ANTHROPIC_* de DeepSeek en esta shell.
ds offLas quita (unset). Claude Code vuelve a tu login.
ds run [cmd]Enciende, ejecuta cmd (o claude), y restaura el estado previo al terminar.
§ 05

Autocompletado

Tab-complete para dsctl y ds: subcomandos, sub-subcomandos (include/exclude/test…), llaves de config y rutas. Se auto-carga al instalar el shell init — normalmente no haces nada.

Si lo quieres cargar a mano (o en otro rc):

# zsh
$ eval "$(dsctl completion zsh)"

# bash
$ eval "$(dsctl completion bash)"

Verifica que cargó con dsctl doctor (línea «Autocompletado cargado»).

§ 06

Integración / scripting

Salida machine-readable para statuslines, hooks y CI.

Exit codes

dsctl resolve y dsctl path test imprimen el veredicto y setean el exit code: 0 = deepseek, 1 = official.

$ dsctl resolve ~/work/proyecto
deepseek
$ echo $?
0

# úsalo en un condicional
$ if dsctl path test . ; then echo "aquí va DeepSeek"; fi

JSON

$ dsctl status --json
{"mode":"deepseek","base_url":"https://api.deepseek.com/anthropic",
 "model":"deepseek-v4-pro[1m]","key_present":true,
 "cwd":"/Users/tu/work","rule":"deepseek","repo":"/Users/tu/work"}

# ejemplo: badge en una statusline
$ dsctl status --json | jq -r '.mode'
CampoValor
modedeepseek | official — estado de esta terminal
ruleVeredicto de la regla para el cwd (independiente de mode)
base_url, modelEndpoint y modelo activos
key_presenttrue/false — si hay API key guardada
cwd, repoCarpeta actual y raíz del repo git (vacío si no es git)

§ 08

Configuración

Modelos y endpoint son configurables. Valores por defecto:

$ dsctl config show
 Base URL:        https://api.deepseek.com/anthropic
 Modelo (pro):    deepseek-v4-pro[1m]
 Modelo (flash):  deepseek-v4-flash
 Effort:          max
$ dsctl config set model_pro deepseek-v4-pro[1m]
$ dsctl config set base_url https://api.deepseek.com/anthropic
$ dsctl config reset          # vuelve a los defaults
LlaveMapea a
base_urlANTHROPIC_BASE_URL
model_proANTHROPIC_MODEL + Opus + Sonnet
model_flashHaiku + CLAUDE_CODE_SUBAGENT_MODEL
effortCLAUDE_CODE_EFFORT_LEVEL
§ 09

Cómo funciona

Un binario corre en un proceso hijo y no puede alterar el entorno del shell padre. Por eso dsctl se divide en tres piezas:

función shell ds on/off/run

Vive en tu rc (la instala dsctl install). Como corre dentro de la shell, es la única que puede export/unset las variables ANTHROPIC_*.

↕ invoca

binario dsctl

Toda la lógica: reglas, key, config, menú, resolución de paths. No toca el entorno; solo lee/escribe config y resuelve veredictos.

↕ consulta cada prompt

hook _ds_autocheck

Se engancha a precmd (zsh) / PROMPT_COMMAND (bash). Llama a dsctl _resolve "$PWD" y enciende/apaga según el veredicto. Cachea por $PWD: solo re-consulta al cambiar de carpeta, así no hay fork en cada prompt.

Tras editar reglas a mano (path edit), haz cd . o re-source tu rc para invalidar el cache del hook.
§ 10

Seguridad

  • La API key vive en ~/.config/dsctl/api_key con permisos 600 — nunca en el rc ni en git.
  • dsctl status solo muestra los últimos 4 caracteres de la key.
  • Las reglas viven en ~/.config/dsctl/rules.tsv (texto plano editable).
  • dsctl nunca exporta variables de forma global: el aislamiento por terminal/carpeta protege tu login oficial.
§ 11

Diagnóstico

$ dsctl doctor

Verifica Node.js, Claude Code, git, la librería de paths, la API key, la función ds y el autocompletado.

SíntomaCausa / arreglo
«Función 'ds' NO cargada»Corre dsctl install y luego source tu rc.
El modo no cambia al hacer cdEl hook no está activo: re-source el rc. Tras editar reglas, cd . para refrescar el cache.
«No hay API key»dsctl key para guardarla.
El autocompletado no respondeRe-source el rc; en zsh requiere compinit.
§ 12

Desinstalar

$ dsctl uninstall                          # quita la función del rc
$ rm -f ~/.local/bin/dsctl ~/.local/lib/dsctl/paths.sh
$ rm -rf ~/.config/dsctl                    # borra key, reglas y config
