# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`ccp` is a Bash CLI that routes Claude Code to a named **profile** per terminal and per directory — never global. A profile is one of: an **official** Anthropic account (its own `CLAUDE_CONFIG_DIR`), a **DeepSeek/compatible provider** (its own `ANTHROPIC_BASE_URL`/`ANTHROPIC_AUTH_TOKEN` + models), or the reserved **`default`** (the user's normal `~/.claude` login). So repo A → account *work*, repo B → account *personal*, repo C → *deepseek*. User-facing strings are in Spanish.

`ccp` supersedes the original `dsctl` (binary toggle deepseek↔official). `bin/dsctl` + `lib/paths.sh`'s old API are gone from the new path; the first `ccp` run auto-migrates `~/.config/dsctl` (creating a builtin `deepseek` profile, converting `include`→deepseek / `exclude`→default). The old rc block is removable via `dsctl uninstall`.

## Commands

```bash
shellcheck -S warning bin/ccp lib/*.sh install.sh tests/run.sh   # lint (CI gate, non-blocking)
bash -n bin/ccp && bash -n lib/paths.sh && bash -n lib/profiles.sh && bash -n lib/env.sh
bash tests/run.sh                                                # full test suite (CI gate)
```

`tests/run.sh` is a hand-rolled harness: it sources the libs, runs every `test_*` function (optional name filter: `bash tests/run.sh resolve`), and exits non-zero on any failure. The lib functions take explicit args (`ccp_home`, rules-file) so tests never touch real config in `~/.config/ccp`. Always run the binary in tests with a temp `CCP_HOME` (an existing dir) so auto-migration does not fire against real `~/.config/dsctl`.

## Architecture

The binary/shell-function split is the central design constraint:

- **`bin/ccp`** — the CLI binary. Owns dispatch, profiles, rules, config, menu, migration, install. Runs in a child process, so it **cannot** mutate the parent shell's environment. Sources three libs (next to itself or from `~/.local/lib/ccp`).
- **Shell function `ccp`** — generated by `_print_shell_init()` and appended to the rc by `ccp install`. Because it runs *in* the shell, only it can `export`/`unset`. It intercepts `use <perfil>` / `default` / `off` / `on` (legacy) / `run`, applying the env delta via `eval "$(command ccp _env <perfil>)"`; everything else falls through to `command ccp "$@"`.
- **`_ccp_autocheck` hook** — generated into the rc; runs on every prompt (zsh `precmd_functions` / bash `PROMPT_COMMAND`), caches by `$PWD`, and applies the resolved profile via `eval "$(command ccp _hook "$PWD")"`. This is why `cd` alone flips the profile.

So: env mutation lives in the rc-installed function; all logic lives in the binary; the hook bridges them by polling `_hook` per directory. The binary EMITS env (`ccp _env`/`_hook`); the shell EVALs it — the only place env changes.

### Libraries

- **`lib/paths.sh`** — pure resolver. `ccp_resolve <path> <rules_file>` returns the **profile name** for a path: among rules whose path is P or an ancestor, the **most specific** (deepest) wins; no match → `default`. `ccp_rule_set`/`ccp_rule_del` mutate `rules.tsv` (`/abs/path<TAB>profile`). `ccp_norm_path` resolves `.`/`..`/`~` textually (not `realpath`) so non-existent paths still work.
- **`lib/profiles.sh`** — storage CRUD on an explicit `ccp_home` arg. `meta` files are `key=value`, **parsed not sourced** (no code execution from config). Functions: `ccp_profile_add_official/add_deepseek/exists/type/get/set_key/get_key/list/rm`.
- **`lib/env.sh`** — `ccp_env_delta <ccp_home> <profile>` emits an eval-able delta: always `unset` all managed vars first, then `export` the target's vars. Every interpolated value is `%q`-quoted (eval-injection safe). Managed vars: `CLAUDE_CONFIG_DIR ANTHROPIC_BASE_URL ANTHROPIC_AUTH_TOKEN ANTHROPIC_MODEL ANTHROPIC_DEFAULT_{OPUS,SONNET,HAIKU}_MODEL CLAUDE_CODE_SUBAGENT_MODEL CLAUDE_CODE_EFFORT_LEVEL CCP_PROFILE`.

### Config & state locations (`~/.config/ccp`)

- `profiles/<name>/meta` — `type=...` + provider fields (`base_url`, `model_pro`, `model_flash`, `effort`).
- `profiles/<name>/api_key` — provider key, `chmod 600`, never in rc or git.
- `profiles/<name>/cc-home/` — official profiles only = that account's `CLAUDE_CONFIG_DIR`. Seeded by `_seed_official_home`: symlinks `plugins/ commands/ agents/ skills/ CLAUDE.md` from `~/.claude` (override `CCP_CLAUDE_SRC`), **copies** `settings.json` (not symlinked — CC writes to it, must not bleed across accounts). The user runs `ccp profile login <name>` once per official profile.
- `profiles.tsv` — index (`name<TAB>type`). `rules.tsv` — path rules. `config` — defaults for new deepseek profiles (parsed, not sourced).

`install.sh` copies `bin/ccp` → `~/.local/bin` and the three libs → `~/.local/lib/ccp`; the binary finds its libs next to itself or falls back to that installed path.

## Conventions

- Commands dispatch through `main()` → `cmd_*`. Internal commands (hook/scripting): `_resolve`, `_env`, `_hook`, `completion-shellinit`. `_migrate_if_needed` runs at the top of `main()` (fires only when `~/.config/dsctl` exists and `CCP_HOME` does not).
- Output helpers `ok/warn/err/info/say/hr` respect `NO_COLOR` and non-TTY. Keep user-facing text Spanish.
- When editing the `ccp` function or hook, edit the heredoc inside `_print_shell_init()` — that text *is* what gets installed. `cmd_uninstall` strips it by matching `# >>> ccp shell init >>>` / `# <<<` markers; don't change those markers casually.
- **macOS portability**: BSD coreutils. Do NOT use `grep -P`; filter with `awk -F'\t'` or `while IFS=$'\t' read`. The `stat` mode check in tests handles both BSD (`stat -f '%Lp'`) and GNU (`stat -c '%a'`).
- **Never pipe `ccp` output into early-closing consumers** (`grep -q`, `head`) in tests — with `set -o pipefail` that surfaces SIGPIPE as a failure. Capture into a var and `case`-match instead. Do NOT add `trap '' PIPE` / `|| true` workarounds to the binary.
- Machine-readable surface: `ccp resolve [path]` and `ccp path test [path]` print the profile name and set exit codes (0=non-default, 1=default); `ccp status --json` emits an object with `active`/`profile`/`profile_type`/`cwd`/`repo`. Keep these stable.
